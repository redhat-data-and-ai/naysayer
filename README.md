# NAYSAYER - Dataproduct Config Review Bot

A self-service GitLab webhook for automatically reviewing warehouse size changes in dataproduct configurations.

## Purpose

NAYSAYER helps the data platform team by automatically approving merge requests that only **decrease** warehouse sizes in `product.yaml` files, while requiring manual review for increases.

**Self-Service Rules:**
- ‚úÖ **Warehouse size decrease** (LARGE ‚Üí SMALL) ‚Üí Auto-approve
- üö´ **Warehouse size increase** (SMALL ‚Üí LARGE) ‚Üí Platform approval needed
- üö´ **No warehouse changes** ‚Üí Standard review process

### Dataproduct YAML Format

```yaml
name: your-dataproduct
kind: source-aligned  # or aggregated
rover_group: dataverse-source-your-dataproduct
warehouses:
  - type: user
    size: XSMALL          # ‚Üê NAYSAYER analyzes this
  - type: service_account
    size: LARGE           # ‚Üê and this
```

### Approval Logic

- **Auto-approve**: Warehouse size decreases only
  - `X6LARGE(10) ‚Üí X5LARGE(9)` ‚úÖ
  - `X5LARGE(9) ‚Üí X4LARGE(8)` ‚úÖ
  - `X4LARGE(8) ‚Üí X3LARGE(7)` ‚úÖ
  - `X3LARGE(7) ‚Üí XXLARGE(6)` ‚úÖ
  - `XXLARGE(6) ‚Üí XLARGE(5)` ‚úÖ
  - `XLARGE(5) ‚Üí LARGE(4)` ‚úÖ
  - `LARGE(4) ‚Üí MEDIUM(3)` ‚úÖ
  - `MEDIUM(3) ‚Üí SMALL(2)` ‚úÖ
  - `SMALL(2) ‚Üí XSMALL(1)` ‚úÖ

- **Require approval**: Any increase or no warehouse changes
  - `SMALL(2) ‚Üí MEDIUM(3)` ‚ùå (platform approval needed)
  - No warehouse changes ‚ùå (standard review process)

## How It Works (Technical Details)

### File Analysis Process

1. **Webhook received** ‚Üí Extract project ID and MR IID
2. **Fetch file changes** ‚Üí Call GitLab API `/projects/:id/merge_requests/:iid/changes`
3. **Analyze config files** ‚Üí Look for warehouse changes in YAML/JSON files
4. **Check diff patterns** ‚Üí Find `-  warehouse: LARGE` ‚Üí `+  warehouse: SMALL`
5. **Make decision** ‚Üí Auto-approve only if all changes are decreases

### Supported File Types

- `.yaml` and `.yml` files
- `.json` files
- Looks for `warehouse:` configuration changes

### Example File Change

The bot analyzes diffs like this:

```diff
# config/dataproduct.yaml
- warehouse: LARGE
+ warehouse: SMALL
```

**Result:** ‚úÖ Auto-approved (decrease detected)

```diff
# config/dataproduct.yaml
- warehouse: SMALL  
+ warehouse: LARGE
```

**Result:** üö´ Requires approval (increase detected)

## Why File-Based Analysis?

**Previous Approach:** Analyzed MR titles for patterns like "Warehouse from LARGE to SMALL"
- ‚ùå Unreliable (depends on title format)
- ‚ùå Easy to bypass
- ‚ùå No validation of actual changes

**Current Approach:** Analyzes actual file diffs for warehouse configuration changes
- ‚úÖ **Accurate** - sees real file changes
- ‚úÖ **Secure** - can't be bypassed with clever titles
- ‚úÖ **Reliable** - works regardless of MR title format
- ‚úÖ **Detailed** - knows which files changed and how
